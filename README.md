# Интернет-магазин «Web-Larёk»
«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и  отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component
Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`


#### Класс Api
Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter
Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` -  хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.

#### Данные
В приложении используются две основные сущности для описания данных:

### Интерфейс товара (IProduct)
Описание товара, доступного для покупки в магазине.

interface IProduct {
  id: string;          // Уникальный идентификатор товара
  description: string; // Подробное описание товара
  image: string;       // URL изображения товара
  title: string;       // Название товара
  category: string;    // Категория товара
  price: number | null; // Цена товара (может быть null)
}

### Интерфейс покупателя (IBuyer)
Данные покупателя, необходимые для оформления заказа.

interface IBuyer {
  payment: TPayment; // Способ оплаты
  email: string;     // Адрес электронной почты
  phone: string;     // Номер телефона
  address: string;   // Адрес доставки
}

#### Модели данных
### Класс ProductList (Каталог товаров)
Назначение и зона ответственности:
Отвечает за хранение и управление списком всех доступных товаров в магазине, а также за хранение товара, выбранного для подробного просмотра.
Конструктор:
`constructor(products?: IProduct[])`
products: IProduct[] (опционально) - начальный массив товаров

Поля класса:
_items: IProduct[] - хранит массив всех товаров каталога
_selectedItem: IProduct | null - хранит товар, выбранный для подробного отображения

Методы класса:
setItems(items: IProduct[]): void - сохраняет массив товаров, полученный в параметрах метода
getItems(): IProduct[] - возвращает массив всех товаров из модели
getItem(id: string): IProduct | undefined - возвращает товар по его id
setSelectedItem(item: IProduct): void - сохраняет товар для подробного отображения
getSelectedItem(): IProduct | null - возвращает товар для подробного отображения

### Класс Cart (Корзина)
Назначение и зона ответственности:
Управляет товарами, выбранными покупателем для покупки. Отвечает за добавление, удаление, подсчет стоимости и количества товаров в корзине.

Конструктор:
`constructor(items?: IProduct[])`
items: IProduct[] (опционально) - начальный массив товаров в корзине

Поля класса:
_items: IProduct[] - хранит массив товаров, находящихся в корзине

Методы класса:
getItems(): IProduct[] - возвращает массив товаров, которые находятся в корзине
addItem(item: IProduct): void - добавляет товар, полученный в параметре, в массив корзины
removeItem(item: IProduct): void - удаляет товар, полученный в параметре, из массива корзины
clear(): void - полностью очищает корзину
getTotalPrice(): number - возвращает общую стоимость всех товаров в корзине
getTotalCount(): number - возвращает общее количество товаров в корзине
contains(id: string): boolean - проверяет наличие товара в корзине по его id

### Класс Buyer (Покупатель)
Назначение и зона ответственности:
Хранит и управляет данными покупателя, необходимыми для оформления заказа. Обеспечивает валидацию данных и их сохранение.

Конструктор:
`constructor(data?: Partial<IBuyer>)`
data: Partial<IBuyer> (опционально) - начальные данные покупателя

Поля класса:
_payment: TPayment - способ оплаты
_email: string - адрес электронной почты
_phone: string - номер телефона
_address: string - адрес доставки

Методы класса:
setData(data: Partial<IBuyer>): void - сохраняет данные покупателя (можно передать как все данные, так и отдельные поля)
setPayment(payment: TPayment): void - сохраняет способ оплаты
setEmail(email: string): void - сохраняет email
setPhone(phone: string): void - сохраняет номер телефона
setAddress(address: string): void - сохраняет адрес доставки
getData(): IBuyer - возвращает все данные покупателя
clear(): void - очищает все данные покупателя
validate(): IBuyerValidationResult - выполняет валидацию данных и возвращает объект с информацией об ошибках

### Интерфейс для результатов валидации:


interface IBuyerValidationResult {
  payment?: string;  // Текст ошибки для способа оплаты
  email?: string;    // Текст ошибки для email
  phone?: string;    // Текст ошибки для телефона
  address?: string;  // Текст ошибки для адреса
}
Метод validate() возвращает объект, где присутствуют только те поля, которые содержат ошибки валидации. Если поле валидно, оно отсутствует в возвращаемом объекте. Поле считается невалидным, если оно пустое.

#### Презентер

# Презентер — это центральный слой приложения, который отвечает за связь между данными (Model) и интерфейсом (View).
В проекте используется один презентер, реализованный в файле main.ts.

Его задача — обработка всех событий, поступающих от моделей и представлений, и принятие решений о том, как изменить состояние приложения или интерфейса.
Презентер не генерирует события, а только слушает их.

#### Основные задачи презентера
### 1. Управление жизненным циклом приложения
загрузка товаров с сервера через ApiClient;
сохранение каталога в модели ProductList;
первичная отрисовка интерфейса.

# 2. Обработка событий моделей

Презентер подписывается на события, которые генерируют модели данных:
catalog:changed → перерисовать каталог;
product:selected → открыть модальное окно с предпросмотром товара;
cart:changed → обновить корзину, каталог и открытое модальное окно;
buyer:changed → обновить обе формы оформления заказа.
Каждый раз при изменении моделей View-компоненты получают актуальные данные через методы вроде renderCatalog(), renderCart(), renderPaymentForm().

### 3. Обработка событий представлений

Представления генерируют события при взаимодействии пользователя. Презентер:
не изменяет DOM напрямую;
не принимает визуальных решений;
всегда вызывает методы моделей или открывает соответствующий View-компонент.

### 4. Управление модальными окнами

Все модальные окна управляются презентером через компонент Modal:
modal.open(viewElement)
modal.setContent(viewElement)
modal.close()
Презентер решает, когда и какое содержимое должно появиться в модалке:
предпросмотр товара;
корзина;
форма оплаты;
форма контактов;
окно успешного заказа.

### 5. Взаимодействие с сервером

Только презентер работает с API:
загрузка каталога: apiClient.getProductList();
отправка заказа: apiClient.submitOrder(orderData).
После успешной отправки заказа:
корзина очищается через модель Cart;
данные покупателя очищаются в Buyer;
открывается модалка с компонентом Success.

### 6. Презентер не хранит данных
Все данные живут только в моделях:
ProductList — каталог и выбранный товар;
Cart — корзина;
Buyer — данные покупателя.
Презентер только:
вызывает методы моделей;
запрашивает у них данные;
передаёт данные в представления.

### 7. Презентер не изменяет DOM

Вся работа с DOM выполняется только компонента View:
Catalog
ProductCard
ProductPreview
Basket
BasketItem
OrderPaymentForm
OrderContactsForm
Header
Modal
Success
Презентер только вызывает:

catalogView.render(...)
basketView.render(...)
productPreviewView.render(...)
orderPaymentView.render(...)
orderContactsView.render(...)
modal.open(...)
modal.setContent(...)

### 8. Ограниченный набор ситуаций для перерисовки View

По требованиям MVP:

Представление перерисовывается только в двух случаях:
— при событии от модели;
— при открытии модального окна.

Это соблюдено:
catalog:changed → renderCatalog();
cart:changed → renderCart(), renderCatalog(), update preview;
buyer:changed → renderPaymentForm(), renderContactsForm();
modal.open() → рендер выбранного компонента.

### 9. Презентер реализует toggle-логику корзины

При клике в карточке товара:

events.on('product:add-to-cart', ({ id }) => {
  if (product.price === null) return;

  if (cart.contains(id)) {
    cart.removeItem(product);
  } else {
    cart.addItem(product);
  }
});


кнопка «В корзину» → «Удалить из корзины»;
если товар бесценный — «Недоступно» (решает View), а презентер просто игнорирует добавление.

### 10. Презентер управляет всей логикой оформления заказа

переход между формами;
обновление ошибок;
проверка валидации;
сбор итогового объекта IOrderData;
отправка на сервер.
